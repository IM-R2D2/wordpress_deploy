# WordPress: автоматическое развёртывание (стейдж / демо)

Скрипт **`deploy.sh`** поднимает на чистой **Debian/Ubuntu** стек **Nginx + MariaDB + PHP-FPM + WordPress** с **WP-CLI**. Рассчитан на тестовые ВМ (в т.ч. **Proxmox**), показ заказчику и отладку до выхода в продакшен.

## Требования

- **ОС:** Ubuntu 22.04 / 24.04 LTS или Debian 12+ (другие дистрибутивы — без гарантий).
- Права **root** на машине (`sudo` или вход под root).
- Доступ в интернет для `apt` и загрузки WordPress / WP-CLI.

## Быстрый старт

1. Скопируйте проект на сервер (или клонируйте репозиторий).
2. Создайте файл окружения:

   ```bash
   cp .env.example .env
   ```

3. Отредактируйте **`.env`**: как минимум **`DOMAIN`** (имя хоста, по которому открываете сайт), **`WP_ADMIN_EMAIL`**, при необходимости **`WP_TITLE`**, **`WP_LOCALE`** (например `fr_CA`).
4. Запустите деплой:

   ```bash
   chmod +x deploy.sh
   sudo ./deploy.sh
   ```

Если **`.env`** нет, используются значения по умолчанию из скрипта (см. комментарии в **`deploy.sh`**). По умолчанию режим **`full`** (стек + один сайт за один запуск).

### Несколько сайтов на одной ВМ

**Общий стек один:** один процесс **Nginx**, один **PHP-FPM**, один сервер **MariaDB**. **Разделяются** инстансы WordPress по трём осям:

| Ось | На практике |
|-----|-------------|
| **Файлы** | Отдельный каталог на сайт (по умолчанию для `site`: `/var/www/sites/ваш-домен`) |
| **База** | Отдельная БД и пользователь (имя БД по умолчанию: `wp_` + нормализованный домен) |
| **HTTP** | Отдельный `server` в Nginx: `server_name` = ваш FQDN, свой корень `root`, отдельные логи и TLS-сертификат (самоподписанный на домен) |

**Шаги:**

1. **Один раз на сервер** (после чистой ОС):

   ```bash
   sudo ./deploy.sh bootstrap
   ```

2. **Каждый новый домен** — отдельный вызов с другим `DOMAIN` (удобно отдельный `.env` на сайт или переменные в одной строке):

   ```bash
   sudo DEPLOY_ACTION=site DOMAIN=client-a.example.com \
     WP_TITLE="Client A" WP_ADMIN_EMAIL="a@example.com" \
     ./deploy.sh site
   ```

   При необходимости первого своего vhost отключите дефолтный сайт Nginx:

   ```bash
   sudo REMOVE_NGINX_DEFAULT=1 DEPLOY_ACTION=site DOMAIN=... ./deploy.sh site
   ```

   Учётные данные по умолчанию пишутся в **`/root/wp-credentials-<slug>.txt`** (slug из домена, например `client-a-example-com`), если не задан **`CREDS_FILE`**.

**Разделение на уровне `.env`:** за один запуск читается **один** файл окружения. Для нескольких сайтов храните **отдельный файл на каждый домен** (например `env/client-a.env`, `env/client-b.env`) и указывайте его через **`DEPLOY_ENV_FILE`**. В каждом файле — свой **`DOMAIN`**, свои **`WP_ADMIN_USER`**, **`WP_ADMIN_EMAIL`**, при необходимости свой **`WP_ADMIN_PASS`** (или пусто → сгенерируется), свой **`CREDS_FILE`**, при желании явные **`DB_NAME`** / **`WEB_ROOT`**. Так у каждого клиента свой логин в админку WP и своя сводка паролей.

Режим **`full`** по-прежнему делает «всё сразу» для одного сайта (как в первой версии скрипта).

### Альтернативный путь к `.env`

```bash
sudo DEPLOY_ENV_FILE=/path/to/.env ./deploy.sh
```

## Что устанавливается

| Компонент | Назначение |
|-----------|------------|
| **Nginx** | Веб-сервер, виртуальный хост под WordPress |
| **MariaDB** | База данных |
| **PHP-FPM** | PHP с расширениями для WordPress (`mysql`, `xml`, `mbstring`, `curl`, `zip`, `gd`, `intl`, `opcache`, `imagick`) |
| **WP-CLI** | Установка ядра WP из командной строки |

Версии **PHP** и **MariaDB** соответствуют репозиторию вашей ОС.

## Конфигурация (`.env`)

Полный список переменных и комментарии — в **[`.env.example`](.env.example)**. Файл подключается через `source` в bash: значения с **пробелами, скобками `()`, восклицательным знаком** и похожими символами указывайте в **двойных кавычках**, например `WP_TITLE="Site WordPress (staging)"`.

Кратко:

| Переменная | Описание |
|------------|----------|
| `DOMAIN` | Имя сервера в Nginx и базовый URL WordPress |
| `WEB_ROOT` | Каталог файлов сайта |
| `DB_NAME`, `DB_USER`, `DB_PASS` | База и пользователь MariaDB; пустой `DB_PASS` → автогенерация |
| `WP_TITLE`, `WP_ADMIN_USER`, `WP_ADMIN_PASS`, `WP_ADMIN_EMAIL` | Первичная установка WP; пустой `WP_ADMIN_PASS` → автогенерация |
| `WP_LOCALE` | Локаль загрузки ядра (необязательно) |
| `SKIP_SSL` | `1` — только HTTP, без самоподписанного HTTPS |
| `SELF_SIGNED_DAYS` | Срок самоподписанного сертификата (если HTTPS включён) |
| `CREDS_FILE` | Куда записать сводку URL и паролей после установки |
| `DEPLOY_ACTION` | `full` \| `bootstrap` \| `site` (можно задать и первым аргументом: `./deploy.sh site`) |
| `SITE_SLUG` | Имя для файлов Nginx/SSL; по умолчанию выводится из `DOMAIN` |
| `REMOVE_NGINX_DEFAULT` | `1` — убрать `sites-enabled/default` (часто для первого `site`) |
| `FORCE_SITE_REINSTALL` | `1` — разрешить установку поверх существующего `wp-config.php` (опасно) |

Файл **`.env`** в репозиторий не попадает — см. **[`.gitignore`](.gitignore)**.

## После установки

- Учётные данные сохраняются в файл, заданный **`CREDS_FILE`** (по умолчанию `/root/wordpress-staging-credentials.txt`, права **600**). Сохраните копию и при возможности удалите файл с сервера.
- По умолчанию включена опция WordPress **«Попросить поисковые системы не индексировать сайт»** (`blog_public = 0`). Для закрытого показа клиенту дополнительно используйте **VPN**, **HTTP Basic Auth** или **файрвол**.
- При **`SKIP_SSL=0`** используется **самоподписанный** сертификат: браузер покажет предупреждение — это нормально для стейджа.
- В продакшене: смена URL в базе, нормальный SSL (например [Let’s Encrypt](https://letsencrypt.org/)), снятие запрета индексации, перенос по стандартной схеме WordPress (дамп БД + `wp-content`).

## Повторный запуск

- **`bootstrap`** повторно гонять обычно не нужно (обновит пакеты).
- **`site`** / **`full`** для **того же** `WEB_ROOT` или той же БД приведёт к ошибке или перезаписи — не запускайте дважды с одинаковыми путями/именами БД без осознанного удаления данных.
- Для нового сайта на той же ВМ используйте **`site`** с **другим** `DOMAIN` (и при необходимости явными **`WEB_ROOT`**, **`DB_NAME`**).

## Безопасность

- Не коммитьте **`.env`** с реальными паролями.
- Ограничьте доступ к стейджу по сети; не публикуйте URL с секретами в открытых каналах.

## Лицензия и ответственность

Скрипт поставляется «как есть». Перед использованием на продакшен-серверах проверьте настройки безопасности, бэкапы и политику вашей организации.
